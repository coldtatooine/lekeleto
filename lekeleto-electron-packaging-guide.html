Tamanho real do instalador

Empacotando tudo, o instalador vai ser grande. É importante ter isso claro antes de começar.

Componente Tamanho Estratégia
Electron runtime (Chromium + Node) ~180 MB Inevitável. É o custo do Electron.
React app + Three.js + deps ~15 MB Bundle otimizado com Vite.
Blender 4.x portable (zip) ~450 MB Versão portable sem Python standalone. Só o essencial.
FFmpeg win64 static ~80 MB Ou ~25 MB com ffmpeg-static (NPM package).
SQLite (better-sqlite3) <2 MB Bundle nativo pelo electron-builder.
Total instalado (aprox.) ~720 MB Instalador NSIS comprimido: ~400 MB
⚠ Consideração importante

O Blender completo inclui assets, templates e add-ons que você não vai usar. Considere uma build customizada do Blender removendo o que não é necessário, reduzindo para ~200 MB. A Blender Foundation documenta como fazer isso via scripts de build.

Resolução de paths: dev vs produção

O maior problema de apps Electron com binários externos é que os paths mudam completamente entre desenvolvimento e produção. Esse módulo centraliza isso.

// src/main/binary-paths.ts
import { app } from 'electron'
import path from 'path'

const isDev = !app.isPackaged

function resourcesPath(): string {
  if (isDev) {
    // Em dev: pasta resources/ na raiz do projeto
    return path.join(__dirname, '../../resources')
  }
  // Em produção: process.resourcesPath aponta para
  // AppData/Local/Lekeleto/resources (após instalação)
  return process.resourcesPath
}

export const BLENDER_PATH = path.join(
  resourcesPath(), 'blender', 'blender.exe'
)

export const FFMPEG_PATH = path.join(
  resourcesPath(), 'ffmpeg', 'ffmpeg.exe'
)

export const SCRIPTS_PATH = path.join(
  resourcesPath(), 'python-scripts'
)

Chamando Blender e FFmpeg via IPC

O renderer (React) nunca chama binários diretamente. Ele envia mensagens IPC para o main process, que executa o child_process.

// src/main/ipc-handlers.ts
import { ipcMain } from 'electron'
import { spawn } from 'child_process'
import { BLENDER_PATH, FFMPEG_PATH, SCRIPTS_PATH } from './binary-paths'
import path from 'path'

ipcMain.handle('render:start', async (event, { sequencePath, outputDir }) => {
  return new Promise((resolve, reject) => {
    const script = path.join(SCRIPTS_PATH, 'render_sequence.py')

    const proc = spawn(BLENDER_PATH, [
      '--background',
      '--python', script,
      '--',                    // args após '--' vão pro script Python
      '--sequence', sequencePath,
      '--output', outputDir,
    ])

    proc.stdout.on('data', (data) => {
      // Envia progresso pro renderer em tempo real
      event.sender.send('render:progress', data.toString())
    })

    proc.on('close', (code) => {
      code === 0 ? resolve({ success: true }) : reject(new Error(`Exit ${code}`))
    })
  })
})

ipcMain.handle('ffmpeg:encode', async (_, { framesDir, outputFile }) => {
  return new Promise((resolve, reject) => {
    const proc = spawn(FFMPEG_PATH, [
      '-framerate', '24',
      '-i', path.join(framesDir, 'frame_%04d.png'),
      '-c:v', 'libx264',
      '-pix_fmt', 'yuv420p',
      '-crf', '18',
      outputFile
    ])
    proc.on('close', (code) =>
      code === 0 ? resolve({ outputFile }) : reject(new Error(`FFmpeg exit ${code}`))
    )
  })
})

Configuração do electron-builder

A chave é a diretiva extraResources, que copia as pastas de binários para dentro do instalador sem incluir no bundle JS.

# electron-builder.yml
appId: com.lekeleto.app
productName: Lekeleto
copyright: Copyright 2025

directories:
  output: dist-installer

files:
  - dist/**/*           # build do Vite/React
  - dist-electron/**/*  # build do main process

extraResources:
  - from: resources/blender
    to: blender
    filter:
      - "**/*"
      - "!**/__pycache__/**"  # exclui cache Python
  - from: resources/ffmpeg
    to: ffmpeg
  - from: resources/python-scripts
    to: python-scripts

win:
  target:
    - target: nsis
      arch: [x64]
  icon: build/icon.ico

nsis:
  oneClick: false          # permite escolher pasta de instalação
  allowToChangeInstallationDirectory: true
  installerIcon: build/icon.ico
  createDesktopShortcut: true
  runAfterFinish: true

SQLite: banco de dados local

Para persistir projetos, assets e histórico de jobs localmente, better-sqlite3 é a escolha certa. Requer rebuild nativo para a versão do Electron.

# package.json — scripts obrigatórios
{
  "scripts": {
    "postinstall": "electron-builder install-app-deps"
  },
  "dependencies": {
    "better-sqlite3": "^9.x"
  }
}
✓ Por que better-sqlite3

API síncrona (ideal pro main process), zero overhead de servidor, arquivo único no AppData do usuário. Perfeito para armazenar projetos, sequências e metadados de jobs.

Checklist de setup do projeto

  • Baixar Blender 4.x portable (.zip, não installer) e extrair em resources/blender/ blender.org/download → Portable (.zip) → Windows x64
  • Baixar FFmpeg static build para Windows ffmpeg.org/download → Windows builds by BtbN → ffmpeg-master-latest-win64-gpl-shared.zip
  • Criar projeto Electron com Vite + React + TypeScript npm create @quick-start/electron ou electron-vite scaffold
  • Instalar e configurar electron-builder com extraResources npm i -D electron-builder
  • Instalar better-sqlite3 e rodar postinstall (rebuild nativo) npm i better-sqlite3 && npm run postinstall
  • Configurar binary-paths.ts com lógica dev/prod Testar os dois cenários antes de avançar no resto
  • Testar spawn do Blender headless com script Python mínimo Confirmar que blender.exe roda via child_process antes de integrar Three.js
  • Testar encode FFmpeg com frames de teste Validar codec h264 e yuv420p antes de conectar ao pipeline

Distribuição: NSIS vs Portable

NSIS Installer

Instala no Program Files, cria atalhos, aparece no "Desinstalar Programas". Experiência profissional. Requer UAC para instalar.

Portable (.zip)

Executa sem instalar. Ideal para testes e distribuição interna. Sem UAC. O usuário descomprime e executa. Mais fácil de iterar no início.

Recomendação: começar com Portable para os primeiros testes internos. Migrar para NSIS quando estiver pronto para distribuição real. O electron-builder suporta ambos simultaneamente.
